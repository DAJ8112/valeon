Section A

The “why,” in my own words

Question

In 4–6 sentences: what does moving this out of a single chat and into a deployed, orchestrated system actually get the business? Focus on the objective and the people, not the technology.

The single chat isn't a viable setup for Valeon - everything lives in one conversation that one person backs up by hand, so the firm's whole capability depends on him. Moving to a real system gives Valeon control over the logic in each step, which can be improved or run independently without affecting the rest. It means more deals handled in parallel, in less time, with senior people spending their hours on judgment and clients instead of driving a chat. For the client, things move faster and the results are more consistent.

Section B

The approach

Question

In broad strokes, how would you restructure this? Breaking up the monolith and what coordinates it; running the whole workflow versus one module; where the human approval points live; and where connectors, sign-in, and storage fit.

01

Breaking the monolith - and what coordinates it

  • Multi-agent system: each agent handles one module's logic.
  • In the center, a single orchestrator agent (a smart, heavy model for better decisions) is the only thing the user talks to - it holds the deal's state and passes structured outputs from one module to the next.
02

The whole workflow versus one module

  • Because the monolith is decomposed, the orchestrator can run the full pipeline or just one part.
  • A single module (e.g., valuation) works standalone because it pulls what it needs from stored deal data, not from a chat history.
03

Human approval points

  • Wherever stakes are high or human judgment is needed (like the thesis check before documents go out), the workflow pauses for human approval.
  • A toggle per run switches between “pause at gates” and fully automated - and who's allowed to flip that toggle is itself permission-controlled.
04

Connectors, sign-in, and storage

  • The whole system sits behind an authorization layer; per account we add integrations (deal CRM, Gmail, more later via OAuth).
  • Persistent storage (relational + vector DB, plus a document store) holds every file and output per deal, so nothing is trapped in a chat.

Section C

Knowledge checks

C1

Improving one module's playbook over time without letting every user change it - and being able to undo a change. What mechanism(s)?

Version control with permissions: a small set of editors can change a playbook, changes are released as tagged versions, and everyone else consumes the released version. Undoing a change is just rolling back to the previous tag.

C2

Run in one chat, the process “drifts.” Why does breaking it into modules reduce that, and what makes a module behave the same each run?

Longer chats degrade outputs because weeks of accumulated, unnecessary information ends up in context window, and the model's outputs starts drifting. Moving away from a single chat, every agent has its own context and access only to what it needs. This makes each module behave consistently each run - we pin its system prompt, keep temperature low, and require a fixed output format.

C3

The deal CRM only supports spreadsheet export/import. How would you design that round-trip so it stays reliable if the workflow is re-run?

Validate the file at both ends of the round-trip - expected columns, no missing or duplicate IDs - so the agents and the CRM always get what they expect. And give every row a permanent unique ID (deal + investor) that the import matches on, updating existing rows instead of adding new ones, so a re-run refreshes the data instead of duplicating it.

C4

One way a multi-step, AI-driven pipeline can fail quietly - and one thing you'd put in place to catch it.

In a multistep pipeline we need to check the outputs of each module because the output of one module will be the input of the next. The outputs might have correct structure but could still be wrong, so even a single bad output can pollute the chain ahead. We need to put observability in the workflow plus simple checks in between so bad output gets flagged at the step that produced it.

Section D

Where I would start

Question

With roughly two weeks and one engineer: what would you ship first, and what would you deliberately leave for later?

My first point of action for these two weeks would be to decompose the workflow from that single chat into a modular workflow. Goal would be simply to connect all these modular agents together and make the whole workflow work just like it did for the 5 clients already with that single chat. Once this core workflow is working only then we can move to making the whole experience better. Things like full automations, other integrations which would be useful, having a good-looking interface for the whole system etc. could be done later. First make it work and then make it better.
